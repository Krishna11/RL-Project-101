## ЁЯЪА Overview

Modern data centers consume **~1тАУ2% of global electricity**, with **30тАУ40% dedicated solely to cooling**. Even small efficiency gains can yield massive cost savings and carbon reductions.

**CoolPilot** explores a powerful idea:

> **Can a Reinforcement Learning agent outperform traditional rule-based cooling systems?**

This project provides an **OpenEnv-compliant RL environment** where an AI agent learns to dynamically control HVAC/CRAC systems in real time.

## ЁЯОп Objective

The agent continuously observes system conditions and adjusts:

| Parameter | Range | Role |
|----------|------|------|
| **Fan Speed** | `0.1 тАУ 1.0` | Controls airflow volume |
| **Chilled Water Flow** | `0.1 тАУ 1.0` | Determines cooling capacity |
| **Supply Air Temperature** | `10┬░C тАУ 20┬░C` | Sets output air temperature |

### Goal:
- Maintain **safe server temperatures (18┬░C тАУ 27┬░C)**  
- Minimize **Power Usage Effectiveness (PUE тЖТ 1.0)**

## ЁЯза Core Concepts

### Power Usage Effectiveness (PUE)

```
PUE = Total Facility Power / IT Equipment Power
```

A PUE of **1.0** is ideal (all power goes to computation). Cooling is the primary driver above 1.0. Our reward signal directly penalizes high PUE.

### Thermal Model (Newton's Law of Cooling)

Each zone's temperature evolves according to:

```
dT/dt = (Q_it - Q_cooling) / (m ├Ч c)
```

| Symbol | Meaning |
|---|---|
| `Q_it` | Heat generated by IT workloads (W) |
| `Q_cooling` | Heat removed by CRAC units (W) |
| `m` | Thermal mass of the zone (kg) |
| `c` | Specific heat capacity (J/kg┬╖K) |

### Reward Function

Each step returns a composite reward in `[0.0, 1.0]`:

| Component | Weight | Description |
|---|---|---|
| **Safety** | 40% | All zones in 18тАУ27┬░C тЖТ 1.0; degrades near boundaries |
| **Energy** | 40% | PUE close to 1.0 тЖТ 1.0; degrades as PUE rises |
| **Stability** | 20% | Penalizes rapid temperature oscillations |

## тЪЩя╕П API Reference

The environment exposes a REST API compatible with the OpenEnv standard:

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/reset` | Start a new episode, returns initial observation |
| `POST` | `/step` | Send an action, returns observation + reward |
| `GET` | `/state` | Current episode metadata |
| `GET` | `/health` | Container liveness check |
| `GET` | `/docs` | Interactive Swagger UI |

### Inference Flow

```
1. POST /reset  тЖТ  Get initial observation
2. Format observation as text prompt
3. Send to LLM (Qwen 72B via hackathon proxy)
4. Parse JSON action from LLM response
5. POST /step with action  тЖТ  Get reward + next observation
6. Log [STEP] to stdout
7. Repeat until done=true
```

The LLM client uses `os.environ["API_BASE_URL"]` and `os.environ["API_KEY"]` directly, with 3 automatic retries and exponential backoff.

### Mandatory Stdout Format

```
[START] task=<task_name> env=coolpilot model=<model_name>
[STEP]  step=<n> action=<action_str> reward=<0.00> done=<true|false> error=<msg|null>
[END]   success=<true|false> steps=<n> score=<0.00> rewards=<r1,r2,...,rn>
```
